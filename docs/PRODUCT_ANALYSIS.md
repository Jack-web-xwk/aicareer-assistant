## 一、现状概述

### 1.1 项目定位

AI 求职助手是一个基于 **FastAPI + LangGraph** 的轻量级 AI 求职工具，面向中国求职者，核心功能涵盖：

1. **简历智能优化** — 上传 PDF/Word 简历，粘贴 Boss 直聘等招聘站岗位链接，AI 基于 STAR 法则生成优化后简历
2. **语音技术面试模拟** — 选择岗位和技术栈后，AI 面试官通过语音实时交互，面试后生成评估报告
3. **职位搜索** — 聚合 Boss 直聘、智联招聘、鱼泡网等多源职位信息
4. **学无止境专栏** — 职业学习相关内容推荐
5. **题库功能** — 技术面试题目浏览

### 1.2 技术栈总览

| 层级 | 技术选型 | 评估 |
|------|---------|------|
| **后端框架** | FastAPI 0.115+ | ✅ 成熟、高性能、异步支持好 |
| **AI 框架** | LangGraph 0.2+, LangChain 0.3+ | ✅ Agent 编排能力强，可观测性好 |
| **LLM 支持** | OpenAI, DeepSeek, 智谱, Ollama, Anthropic, 通义千问, 百炼 | ✅ 多模型切换，架构灵活 |
| **简历解析** | pdfplumber, python-docx | ⚠️ 基础能力，复杂排版可能丢失信息 |
| **数据爬取** | requests, BeautifulSoup4 | ⚠️ 反爬对抗弱，容易被封 |
| **语音处理** | OpenAI Whisper + TTS | ⚠️ 同步调用阻塞事件循环 |
| **数据库** | SQLite + SQLAlchemy 2.0 + aiosqlite | ⚠️ 仅适合原型，不支持并发 |
| **前端** | React 18 + TypeScript + Ant Design 5.x | ✅ 生态成熟，组件丰富 |
| **状态管理** | Zustand | ✅ 轻量、直觉式 API |
| **图表** | Recharts | ✅ 与 Ant Design 风格一致 |
| **动画** | Framer Motion | ✅ 声明式、交互友好 |

### 1.4 架构评估

**优势**:
- 分层清晰：API → Services → Agents → Core/Models
- 多 LLM 支持架构优秀，通过配置即可切换模型
- LangGraph 实现了简历优化的可观测性（节点进度、中间结果）
- CORS、限流、异常处理等基础能力已具备

**不足**:
- 单体架构，无法水平扩展
- SQLite 不适合生产环境
- 会话状态存储在内存中
- 无用户认证系统

## 二、问题清单

### 2.1 🔴 高优先级问题

#### P0-1: 内存会话存储 — 生产环境致命缺陷
**位置**: `backend/app/api/interview.py:35`  
**问题**: `active_sessions: Dict[str, InterviewState] = {}` 使用进程内字典存储所有面试会话  
**影响**:
- 服务器重启 → 所有活跃面试数据丢失
- 无法水平扩展（多实例负载均衡后会话断裂）
- 内存泄漏风险（异常断开的 WebSocket 对应会话不清理）

**已部分修复**: `Pod_D` 阶段创建了 Redis 客户端 (`redis_client.py`)，但在当前代码中 Redis 仍为可选依赖，默认仍回退到内存存储。

#### P0-2: 同步音频处理 — 高延迟阻塞
**位置**: `backend/app/services/audio_processor.py`  
**问题**: Whisper 调用和 TTS 合成均为同步阻塞操作  
**影响**:
- 单次问答周期 P95 延迟 ~8.6 秒
- 单 Worker QPS < 0.2
- 完全阻塞异步事件循环

**部分进展**: `streaming_audio_processor.py`（16KB）实现了流式处理和 Celery 集成，但需要额外部署 Redis + Celery Worker，增加了运维复杂度。

#### P0-3: 无用户认证系统
**问题**: 整个项目没有用户系统，所有数据无归属  
**影响**:
- 无法支持多用户
- 无法做个性化推荐
- 无法做数据持久化和历史追溯

#### P0-4: SQLite 不适合生产
**位置**: `backend/app/core/config.py:DATABASE_URL`  
**问题**: `sqlite+aiosqlite:///./data/career_assistant.db`  
**影响**:
- 无并发写入能力
- 无事务隔离
- 数据文件在服务器本地，备份困难

### 2.2 🟡 中优先级问题

#### P1-1: WebSocket 连接泄漏风险
**位置**: `backend/app/api/interview.py`  
**问题**: 无心跳检测、无连接数限制、无超时保护  
**状态**: `websocket_manager.py`（20KB）已实现心跳和清理机制，但需验证生产稳定性

#### P1-2: 职位爬虫反爬对抗弱
**位置**: `backend/app/services/job_scraper.py`（32KB）  
**问题**: 基于 requests + BeautifulSoup 的简单爬虫，无代理池、无指纹伪装、无自动重试策略  
**影响**: 高频使用后会被各平台封 IP，功能失效

#### P1-3: 测试覆盖率极低
**位置**: `backend/tests/`  
**数据**: 仅 5 个测试用例通过，总体覆盖率约 25%  
**缺口**:
- 简历优化核心功能零测试
- LangGraph 工作流零测试
- API 集成测试缺失
- 前端无任何测试框架

#### P1-4: 前端大型单页组件
**位置**: `frontend/src/pages/InterviewSimulatorPage.tsx`（25KB）  
**问题**: 所有面试逻辑集中在单一组件中，状态耦合严重，难以维护和测试  
**建议**: 拆分为 InterviewPrepPage、InterviewReplayPage、ProgressDashboardPage（部分已完成）

#### P1-5: 简历解析能力有限
**位置**: `backend/app/services/resume_parser.py`（7.5KB）  
**问题**: 
- 不支持扫描版 PDF（无 OCR）
- 复杂排版（多栏、表格内嵌）可能丢失信息
- 无法处理图片格式简历

### 2.3 🟢 低优先级问题

#### P2-1: 无 CI/CD 流水线
无 GitHub Actions 或其他自动化构建/测试/部署配置

#### P2-2: 无监控告警
Prometheus 指标已创建（`metrics/interview_metrics.py`），但无 Grafana 面板或告警规则

#### P2-3: 日志系统基础
仅有 Python logging，无结构化日志、无日志聚合

#### P2-4: 无 API 版本管理
API 路径无 `/v1/` 前缀，未来升级可能产生兼容性问题

#### P2-5: 安全隐患
- config.py 中硬编码了 DeepSeek API Key（`sk-3c374bea34474a5196427330fdfaddd4`）
- CORS 配置过于宽松（允许所有来源的 `*` 风险）
- 无请求体大小限制的全局中间件

## 四、功能差距分析与产品化建议

### 4.1 必须补齐的基础能力（P0）

#### 🔐 用户认证与授权
- **必要性**: 任何商业产品的基础
- **建议方案**: 
  - 短期: JWT Token + 手机号/邮箱注册（国内场景优先手机号）
  - 长期: 对接微信/钉钉/飞书 SSO
- **工作量**: 约 2-3 天

#### 💾 数据库升级
- **必要性**: SQLite 不支持生产并发
- **建议方案**:
  - 开发/测试: SQLite（保持现状）
  - 生产: PostgreSQL（主）+ Redis（缓存/会话）
- **工作量**: 约 3-5 天（含数据迁移脚本）

#### 📊 用户仪表盘
- **必要性**: 提升用户粘性和留存
- **功能**:
  - 面试历史记录与趋势图
  - 简历优化历史
  - 能力雷达图（技术/沟通/逻辑/表达/潜力）
  - 学习进度追踪
- **工作量**: 约 5-7 天（前端 10725 字节已开始，需补充 API）

### 4.2 提升核心竞争力的功能（P1）

#### 🎯 ATS 评分体系
- **必要性**: 求职者最关心的指标之一
- **建议方案**:

- **工作量**: 约 5-7 天

#### 📱 移动端适配
- **建议方案**:

  - 长期: React Native 或小程序
- **工作量**: PWA 约 3-5 天

#### 🎙️ 面试回放与复盘
- **功能**: 面试过程录音回放、逐句复盘、对比两次回答
- **状态**: 前端已有 `InterviewReplayPage.tsx`（15KB），需补充后端 Replay API

#### 📈 进度追踪 Dashboard

- **状态**: 前端已有 `ProgressDashboardPage.tsx`（10KB），需补充后端聚合 API

#### 关键技术决策

| 决策点 | 当前 | 建议 | 理由 |
|--------|------|------|------|
| **数据库** | SQLite | PostgreSQL + Redis | 并发、可靠性、扩展性 |
| **认证** | 无 | JWT + OAuth2 (FastAPI Users) | 安全、标准 |
| **部署** | 本地开发 | Docker + Docker Compose | 一致性、可移植 |
| **前端构建** | Vite dev | Vite build + Nginx + CDN | 性能、缓存 |
| **异步任务** | 无 | Celery + Redis | 音频处理、邮件通知 |
| **文件存储** | 本地磁盘 | MinIO / 阿里云 OSS | 可靠性、CDN 加速 |
| **日志** | Python logging | Structlog + ELK | 可观测性 |
| **监控** | Prometheus 指标 | Grafana Dashboard | 可视化、告警 |

## 七、总结

### 项目当前状态: 🟡 MVP 阶段，有潜力但缺基础

**核心优势**:
1. 技术栈选型合理（FastAPI + LangGraph + React）
2. 多 LLM 支持架构灵活
3. 简历优化 + 语音面试的组合在国内开源项目中稀缺
4. LangGraph 可观测性是差异化亮点
5. 代码结构清晰，模块化程度较好

**核心短板**:
1. 无用户系统 — 无法做任何产品化
2. SQLite 存储 — 无法支撑生产并发
3. 测试覆盖率 25% — 难以保证质量
4. 爬虫方案脆弱 — 职位搜索功能可能随时失效
5. 无安全合规 — API Key 硬编码、无认证、CORS 过宽

### 投资建议
如果目标是商业产品，**Phase 0（基础加固）是必须完成的前提**，预计 1-2 周。在此之后，简历优化 + 面试模拟的组合功能有明确的市场空白（国内缺乏开源的全流程 AI 求职助手），值得继续投入。

如果目标是个人项目/学习用途，当前状态已基本可用，重点改进测试覆盖和用户体验即可。
