# 🧬 AI微调周报 - 03.24-03.30

## 🏆 本周Top进展

### 1. Axolotl v0.15.0：MoE微调迎来里程碑式优化
Axolotl 发布 v0.15.0，三大核心升级：
- **ScatterMoE LoRA + SonicMoE**：为 MoE 模型提供专用 LoRA 支持和更快的训练内核，相比 transformers 的 grouped_mm 显著提升速度和显存效率
- **MoE Expert 量化**：支持 Transformers v5 下的专家权重量化，GLM-4.7-Flash QLoRA 显存从 ~127GiB 暴降至 ~23GiB，降幅达 **82%**
- **Torch 2.10.0 + uv 构建**：更快速、更可复现的 Docker 镜像构建

**影响分析**：MoE 模型（如 Mixtral、Qwen3.5 MoE、GLM-4.7）的微调门槛大幅降低。之前一张 A100 跑不动的 MoE QLoRA，现在单卡可跑，这将加速 MoE 架构在垂直领域的落地。

### 2. Unsloth Studio 加速进化：推理提速 20-30%
3月27日 Unsloth 发布重要更新：
- **推理速度提升 20-30%**：修复了 tool-calling 和 repeat penalty 导致的推理降速问题
- **修复 repetition_penalty 默认值导致的 24% TPS 下降**（关键 bug fix）
- **Gemma3N 音频训练修复**：修复 stride assertion 与 non-reentrant checkpointing 的兼容问题
- **自动检测 reasoning 模型**：自动识别 always-on reasoning 模型并锁定 Think 按钮
- **FP8 强化学习**：与 PyTorch 合作的 FP8 RL 支持，GRPO 训练显存降低 80%

**影响分析**：Unsloth 正从纯微调工具转型为一站式模型训练/推理平台（Studio），Web UI + 数据可视化 + 训练历史追踪，降低了个人开发者的使用门槛。

### 3. MoE-Sieve：路由引导的 LoRA 高效微调（arXiv 3.25）
论文提出 MoE-Sieve，针对 MoE 模型的标准 LoRA 微调低效问题，通过路由信息指导 LoRA 参数分配：
- 不是所有专家都需要同等微调力度
- 利用 router 的激活模式动态分配 LoRA rank
- 在保持性能的同时大幅减少可训练参数

**影响分析**：随着 Qwen3.5 MoE、DeepSeek-V3 等 MoE 架构成为主流，"如何高效微调 MoE" 是当前最核心的工程问题之一。MoE-Sieve 与 Axolotl 的 ScatterMoE LoRA 形成了学术+工程的互补推进。

### 4. Label-Free Cross-Task LoRA Merging（arXiv 3.27）
提出基于零空间压缩的无标签跨任务 LoRA 合并方法：
- 无需额外标注数据即可合并多个 LoRA 适配器
- 通过 null-space 压缩避免任务间的干扰
- 解决多任务 LoRA 合并中的灾难性遗忘问题

**影响分析**：LoRA 合并（model merging）是当前微调生态的热点方向。该方法降低了多任务合并的数据成本，对需要一个模型服务多场景的团队特别有价值。

### 5. TuneShift-KD：微调模型的知识蒸馏新框架（arXiv 3.25）
专门针对已微调模型的知识迁移场景：
- 解决"微调后的专有模型如何低成本迁移到小模型"的问题
- 不同于传统 KD，TuneShift-KD 专注于微调引入的增量知识
- 支持将大模型微调成果蒸馏到更小的部署友好模型

**影响分析**：企业微调的最终目的是部署。大模型微调 → 小模型蒸馏 → 边缘部署，这条路径正在变得更加成熟。

## 💼 落地案例

### 医疗文本摘要：PEFT 方法系统对比
一篇来自多机构的研究系统对比了 LoRA、Prompt Tuning 和 Full Fine-Tuning 在医疗文本摘要任务上的表现：
- LoRA 在大多数场景下以 **<1% 的参数量** 达到了 Full FT 90%+ 的效果
- Prompt Tuning 在低资源场景下表现更稳定
- 建议：高资源选 LoRA，低资源考虑 Prompt Tuning

### XBRL 金融标签：领域微调 + LLM 零样本重排序
金融 XBRL 报告中的标签选择是行业痛点。本周论文 XBRLTagRec 提出：
- 对 LLM 进行领域微调
- 结合零样本重排序
- 在极端金融数字标注任务上取得 SOTA

### 农业大模型：AgriPestDatabase 结构化数据集
面向农业害虫识别的 LLM 微调，通过构建结构化昆虫数据集，微调 LLaMA 3.2 7B 实现农业领域的专业问答能力。

## 📊 趋势洞察

### 1. MoE 微调成为主战场
Qwen3.5 MoE、DeepSeek-V3、GLM-4.7 等 MoE 架构模型密集发布，带动了 MoE 微调技术的爆发。本周同时出现了 Axolotl 的 ScatterMoE LoRA（工程）和 MoE-Sieve（学术），说明这个方向正在被系统性攻克。**核心矛盾**：MoE 模型参数量巨大但推理高效，如何让微调也"高效"是关键。

### 2. LoRA 生态持续分化
LoRA 不再只是"低秩适配"，正在分化为多个子方向：
- **LoRA 合并**（Null-Space Merging）→ 多任务服务
- **自适应 LoRA rank**（Adaptive Ranks）→ 分层精细化
- **MoE LoRA**（MoE-Sieve, ScatterMoE）→ 针对 MoE 架构优化
- **LoRA + 检索增强**（RAG Agent SFT）→ Agent 训练新范式

### 3. 微调工具的"平台化"趋势
Unsloth Studio 的推出标志着微调工具正从 CLI 向 Web UI / 平台化演进：
- 可视化数据准备（Data Recipes）
- 训练监控与历史追踪
- 一键模型导出（GGUF / Safetensors）
- 内置推理 + Tool Calling

这降低了微调的技术门槛，目标用户从 ML Engineer 扩展到更广泛的开发者群体。

### 4. 强化学习微调的工程化成熟
GRPO 正在替代 PPO 成为主流 RLHF 方法，Axolotl 和 Unsloth 都在重点投入：
- Unsloth 的 FP8 GRPO 将显存需求降低 80%
- Axolotl 修复了 GRPO 的多个配置问题
- 预计 Q2 将看到更多 GRPO 在生产环境的落地案例

### 5. 端侧部署驱动蒸馏需求
随着边缘部署需求增长（手机、嵌入式设备），"大模型微调 → 小模型蒸馏" 的 Pipeline 正在标准化。TuneShift-KD 等工作专门针对微调后的蒸馏场景，不再是通用 KD 的简单套用。
